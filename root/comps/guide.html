How to Create Your Backend

Initially, you must make an important decision: Do you require server-side rendering (SSR) or is client-side rendering (CSR) sufficient? Go4lage is already equipped for CSR. For SSR, while a template replacement is available, it may not include all the features you might need.

Begin by designing your SQL data structure. Create a map of the information you intend to store in your backend.

Examine the existing user, group, and permission structure. Avoid altering this and maintain the integrity of the domain. Instead, expand it by using their primary keys as foreign keys in your new tables. This approach aligns with domain-driven design principles.

With a Go backend, you can define and serialize complex queries using Golang. If your data spans multiple tables, you can access it seamlessly and at high speed. Therefore, focus on establishing a clear, simple, and logical structure in the database rather than prematurely concentrating on frontend views.

After completing your data structure, write the SQL schema and queries. SQLC will generate useful functions for you.

Proceed to write your endpoints. Implement your functions in the urls.go file. These functions will respond with JSON (for CSR) or HTML (for SSR).

Go4lage Built-in Commands

The following commands are included in the Go4lage CLI:

  • startserver
  • createsuperuser
  • setupgp
  • rungoose

The startserver and createsuperuser commands are straightforward. The setupgp command will create groups and permissions on the fly, eliminating the need for the developer to manually adjust the admin dashboard.

The rungoose command runs Goose, the database migration tool, with options like "up", "up-by-one", "down", "redo", "reset", "status", "version", and "fix".

Package Management

Go is designed to have several small parts that each do one thing correctly. However, this is not always practical, and Go4lage extends this principle.

Go4lage itself should be the main package, and your endpoints will be in a new package that is imported into the main package.

There is another package, utils, that exists to avoid cyclic dependencies. (You will probably need the UserKey struct from it to get your user from the request context.)

Despite using your own package for your main changes and mostly modifying only urls.go, you still need to use the sql/schema and sql/queries folders. Again, it is recommended not to alter the existing structure and to make only additive (domain-driven) changes.
